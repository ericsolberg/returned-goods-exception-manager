# Setup Solution

The purpose of this skill is to set up the solution structure inside the current working directory, containing a `solution.yaml` and an `assets/` folder with one subfolder per asset, each defined by an `asset.yaml` file.

**All files created by this skill MUST be inside the current directory.**

## Instructions

### Schema Reference

**solution.yaml Structure:**
```yaml
apiVersion: solution.sap/v1
kind: Solution

metadata:
  name: <solution-name>                 # Unique name for your solution
  version: 1.0.0                        # Solution version
  description: <solution-description>   # Brief description of the solution

assets:                                 # Assets to include
  - ref: ./assets/<asset-name>/asset.yaml
```

**asset.yaml Structure:**
```yaml
apiVersion: asset.sap/v1
kind: Asset

metadata:                 # Asset identity - modifying the name after deployment will cause breaking changes (such as URLs of the deployed asset)
  name: <asset-name>      # Identifying name using kebab case (only lower case chars and dashes, max 63 chars)
  version: 1.0.0          # Semantic version of this asset

type: <asset-type>        # Asset type
 
sourceRoot: "workflows"   # Root directory where workflow files are located
 
# Artifacts provided by this asset
provides:
  <artifact type specific content>
```

### Asset Types

- `base-ui`: Frontend web applications (Next.js, React)
- `agent`: AI agent services (LangChain, custom)
- `agent-extension`: Agent extension packages that extend a base agent with additional capabilities
- `n8n-workflow`: N8N workflow automation
- `vercel-app`: Vercel-specific deployments
- `mcp-server`: MCP Server metadata files (i.e. translation files for API-to-MCP tool mapping). **This asset type contains only `asset.yaml`, `translation.json`, and `api-spec.json` — NO server implementation, NO Dockerfile, NO source code.** The translation file is generated by the `mcp-translation-file` skill and copied into the asset folder. The original API spec file used to generate the translation should also be copied for reference.

**Note**: Each asset type has a detailed JSON Schema in this skill's folder (`skills/setup-solution/assets/{type}.json`) with:
- Complete property definitions (100+ properties for agents)
- Strict validation rules (patterns, enums, min/max values)
- Detailed descriptions for every field
- Default values and constraints
- Examples: agent.json includes agentCard, models, secrets, HPA, probes, resources, virtualService, etc.

### Multi-Component Pattern (AppRouter + Next.js)

**IMPORTANT**: For AppRouter + Next.js setups:

1. **Order matters**: AppRouter MUST be first component
2. **Destinations**: Use environment variable, NOT xs-app.json
3. **Ports**: AppRouter = 5000, Next.js = 3000

```yaml
components:
  - name: appRouter
    buildPath: router
    port: 5000
    env:
      destinations: '[{"name":"backend","url":"http://127.0.0.1:3000","forwardAuthToken":false}]'

  - name: server
    buildPath: src
    port: 3000
    env:
      PORT: "3000"
```

**xs-app.json** (minimal, NO destinations block):
```json
{
  "welcomeFile": "/",
  "authenticationMethod": "none",
  "sessionTimeout": 60,
  "routes": [
    {
      "source": "^/(.*)$",
      "target": "$1",
      "destination": "backend",
      "authenticationType": "none"
    }
  ],
  "compression": {"enabled": true},
  "websockets": {"enabled": true}
}
```

### Examples

**Example 1: Simple Solution**
```yaml
apiVersion: solution.sap/v1
kind: Solution
metadata:
  name: my-app
  version: "1.0.0"
assets:
  - ref: ./assets/frontend/asset.yaml
```

**Example 2: Python Agent Asset**
```yaml
apiVersion: asset.sap/v1
kind: Asset             # Must be 'Asset' (capital A)
metadata:
  id: ai-assistant     # Required: used by deployer as asset_id
  name: AI Assistant   # Optional
type: agent
container:
  buildPath: .
  port: 5000
probes:
  startup:
    path: /.well-known/agent.json
    periodSeconds: 5
    timeoutSeconds: 3
    failureThreshold: 18
  liveness:
    path: /.well-known/agent.json
    initialDelaySeconds: 15
    periodSeconds: 10
    timeoutSeconds: 5
    failureThreshold: 3
  readiness:
    path: /.well-known/agent.json
    initialDelaySeconds: 5
    periodSeconds: 5
    timeoutSeconds: 3
    failureThreshold: 3
```

**Example 3: MCP Server Asset**
```yaml
apiVersion: asset.sap/v1
kind: Asset
type: mcp-server
metadata:
  id: s4-dispute-mcp-server
  name: "Dispute MCP Server"
  version: "1.0.0"
  translation: translation.json
```

The MCP server asset folder contains **only** three metadata files:
```
assets/s4-dispute-mcp-server/
├── asset.yaml          # as above
├── api-spec.json       # original API spec copied from openspec/changes/<change-id>/api-specs/
└── translation.json    # copied from openspec/changes/<change-id>/mcps/
```
Do NOT create a Dockerfile, source code, or any other files for MCP server assets.

**Translation file source:** Translation files are generated by the `mcp-translation-file` skill and stored at `openspec/changes/<change-id>/mcps/{mcp-server-name}.translation.json`. When setting up MCP server assets, copy each translation file from that location into its corresponding asset folder, renaming it to `translation.json`. Also copy the original API spec file from `openspec/changes/<change-id>/api-specs/` into the asset folder, renaming it to `api-spec.json`.

**Example 4: Multi-Asset Solution (Agent + MCP Servers)**
```yaml
apiVersion: solution.sap/v1
kind: Solution
metadata:
  name: full-stack-app
  version: "1.0.0"
  description: "Agent + MCP Server example"
assets:
  - ref: ./assets/agent/asset.yaml
  - ref: ./assets/s4-dispute-mcp-server/asset.yaml
```

When an agent needs access to tools defined in MCP server translation files, add the MCP server ORD IDs to the agent's `asset.yaml` under `tools.mcp-servers`:
```yaml
# In assets/agent/asset.yaml:
apiVersion: asset.sap/v1
kind: Asset
type: agent
metadata:
  id: my-agent
# ... other agent config ...
tools:
  mcp-servers:
    - sap.mcp:apiResource:s4-dispute-mcp-server:v1
```
The ORD IDs in `tools.mcp-servers` must match the ORD IDs of the MCP server assets defined in their respective `asset.yaml` files.

### Best Practices

1. **Do NOT add `container.env` to agent assets** — it causes schema validation errors on the platform. Set environment variables via `ENV` instructions in the Dockerfile instead. The `env` block is only valid for `base-ui` / `service` component definitions.

2. **Keep agent `asset.yaml` minimal** — omit `resources` and `hpa` blocks unless you have a specific reason to override platform defaults. Extra fields have caused schema validation failures on the platform.

3. **Include health probes** (use `/.well-known/agent.json` for A2A agents):
   ```yaml
   probes:
     startup:
       path: /.well-known/agent.json
       periodSeconds: 5
       timeoutSeconds: 3
       failureThreshold: 18
     liveness:
       path: /.well-known/agent.json
       initialDelaySeconds: 15
       periodSeconds: 10
       timeoutSeconds: 5
       failureThreshold: 3
     readiness:
       path: /.well-known/agent.json
       initialDelaySeconds: 5
       periodSeconds: 5
       timeoutSeconds: 3
       failureThreshold: 3
   ```

4. **Use environment variables** for configuration (UI/service assets only — see note 1 for agents):
   ```yaml
   env:
     PORT: "3000"
     LOG_LEVEL: info
   ```

5. **For AppRouter assets**, NEVER include destinations in xs-app.json - use env var instead

6. **Standard ports**:
  - Python agents: 5000
  - Node.js: 3000
  - Java: 8080
  - AppRouter: 5000

7. **Required folder structure** — always use this layout:
  ```
  ./
  ├── solution.yaml             # refs ./assets/<asset-name>/asset.yaml
  └── assets/
      └── <asset-name>/
        ├── asset.yaml
        └── ...                 # any other asset files that are needed (e.g. Dockerfile, source code, xs-app.json, translation file, etc.).
  ```
  When applicable:
    - The `Dockerfile` MUST be co-located with `asset.yaml` (i.e. inside `assets/<asset-name>/`).
    - The `container.buildPath` in `asset.yaml` must be `.` (relative to asset.yaml location).

### Common Issues

**AppRouter Fails with "unknown destination backend"**
- ✅ Remove `destinations` block from xs-app.json
- ✅ Add `destinations` environment variable in asset.yaml

**Service exposes wrong port**
- ✅ Ensure AppRouter is first component in list

**Build fails**
- ✅ Check buildPath points to correct directory
- ✅ Verify package.json exists

### Workflow

When user requests YAML creation:

1. All solution files (i.e. `solution.yaml` and `assets/` folder) **MUST** live inside the current working directory. Create the assets folder if it does not exist (`mkdir -p assets`). 

2. **Ask clarifying questions if unclear**:
   - What type of asset? (UI, agent, service, mcp-server)
   - Single or multi-container? (not applicable for mcp-server)
   - What framework? (Next.js, Python, etc.) (not applicable for mcp-server)

3. **Consult JSON schemas for validation rules**:
   - Check `assets/solution-schema.json` for solution structure
   - Check `assets/{type}.json` for asset-specific fields
   - Note required fields, patterns (e.g., DNS-compatible names), and enums

4. **Create appropriate structure inside `assets/`**:
   - Use single-container for simple apps
   - Use multi-component for AppRouter patterns
   - For `mcp-server` assets: create only `asset.yaml` + copy `translation.json` + copy `api-spec.json` — nothing else

5. **Handle MCP server assets** (only do this if translation files exist in `specs/<asset-name>/openspec/changes/<change-id>/mcps/`, otherwise ignore this step):
   - For each `{name}.translation.json` in the mcps folder:
     a. Create the asset folder: `assets/{mcp-server-name}/`
     b. Copy the translation file: `cp specs/<asset-name>/openspec/changes/<change-id>/mcps/{name}.translation.json assets/{mcp-server-name}/translation.json`
     c. Copy the original API spec: `cp specs/<asset-name>/openspec/changes/<change-id>/api-specs/{name}.json assets/{mcp-server-name}/api-spec.json`
     d. Create `assets/{mcp-server-name}/asset.yaml` following the `mcp-server` schema (see Example 3)
     e. Register the asset in `solution.yaml` under `assets`, i.e.: `- ref: ./assets/{mcp-server-name}/asset.yaml`
   - If the solution includes an agent asset that uses these APIs, add the MCP server ORD IDs to the agent's `asset.yaml` under `tools.mcp-servers` (see Example 4 and 5)
   - Do NOT create a Dockerfile, source code, or server implementation for MCP server assets

6. **Include all required fields**:
   - metadata.name (solution) - must match pattern `^[a-z0-9]([-a-z0-9]*[a-z0-9])?$`
   - metadata.id (asset) - DNS-compatible
   - type - must match available schemas
   - container OR components (not applicable for mcp-server assets)

7. **Add best practices**:
   - Resource limits and environment variables (UI/service assets only — omit for agent assets, see Best Practices)
   - Health probes

8. **Validate**:
   - Check YAML syntax
   - Verify all required fields present (per JSON schema)
   - Ensure paths are correct and all paths are relative to `solution.yaml`
   - Validate against patterns and constraints

## Important Notes

- Schema version: v3.0.0
- All paths in `solution.yaml` are relative to the `solution.yaml` location
- Field names are case-sensitive
- Use lowercase-with-hyphens for IDs
- The `Dockerfile` must live inside the asset folder (co-located with `asset.yaml`), not at the solution root
- For A2A agents, always use `/.well-known/agent.json` as the probe path (startup + liveness + readiness)
- MCP server assets contain ONLY `asset.yaml` + `translation.json` + `api-spec.json` — no Dockerfile, no source code, no server implementation
